## Scope completed

pure file generator (no custom integration runtime):

- `render_codex_bundle(project_root) -> dict[str, str]`
- `render_copilot_bundle(project_root) -> dict[str, str]`
- `render_opencode_bundle(project_root) -> dict[str, str]`
- `export_agent_bundle(target, out_dir, *, project_root=None, store=None) -> list[Path]`
- `default_out_dir(project_root, target) -> Path`

Wired the previously-stubbed `dmc export-agent-bundle` CLI command (owned by
M10 per the bootstrap CLI comment) to the new adapter module, and closed the
Path-serialization gap in M09's lazy `dmc_export_agent_bundle` MCP wrapper now
that `dmc.adapters` genuinely exists.

### Revision (PR #3 review, CHANGES_REQUIRED)

Two blocking issues were fixed before merge:

1. The Codex config template referenced an unimplemented `DMC_ROOT` env
   override. Replaced with the real, already-supported `dmc serve --root .`
   CLI flag.
2. `export_agent_bundle`'s `out` parameter defaulted to the resolved project
   root when omitted, which would silently overwrite a caller's own root
   files (e.g. this repo's own `AGENTS.md`) on a codex/opencode export with
   no explicit output directory. Renamed `out` -> `out_dir` (required, no
   default; matches the module card's contract name) and added
   `default_out_dir(project_root, target)` — a `.dmc/adapters/generated/<target>`
   staging path used by the CLI and MCP wrapper whenever the caller does not
   supply an explicit output directory, so a bundle export can never
   silently overwrite root files.

## Known limitations

- `export_agent_bundle` does not merge bundles: if both `codex` and
  `opencode` are exported to the same `out_dir`, the second call's
  `AGENTS.md` overwrites the first's (both files are protocol-identical in
  content today, so this is currently a no-op collision, but a future change
  that diverges the two would need an explicit merge policy).
- No `.codex/`, `.github/skills/`, or `.dmc/adapters/opencode/` directories
  are pre-existing in this repo's own tree; bundle generation is exercised
  only against temp directories in tests, per module scope ("output is
  files/instructions/config templates, not a custom integration runtime" —
  this repo does not consume its own generated bundles).
