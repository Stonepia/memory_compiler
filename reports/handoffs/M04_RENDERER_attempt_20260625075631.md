## Important implementation notes

- Imports all data shapes from `src/dmc/schemas.py` (`PlanGraph`, `PlanNode`,
  `TaskRequest`, `SearchResult`); no shapes redefined. Reuses
  `dmc.planner.next_ready_nodes` to derive briefing next actions.
- `render_plan_mermaid`: output starts with `flowchart TD`; every node label is
  `"<id> (<type>) | <goal> | gates: ..."`; one `dep --> node` edge per
  dependency, in node/dependency order; human-review and evidence gates are
  surfaced as `human_review` / `evidence` markers in node labels. Labels are
  sanitized (`[`/`]`/quotes/newlines stripped) so the flowchart stays valid.
- `render_plan_markdown`: header + per-node section with goal, dependencies,
  explicit `Human review gate: yes/no`, `Evidence gate: yes/no`, and success
  criteria.
- `render_briefing`: always emits the six required sections in order — Selected
  workflows, Atoms, Knowledge refs, Pitfalls, Open questions, Next actions.
  Empty categories render an explicit placeholder line (never omitted). Optional
  `context: list[SearchResult]` is split into pitfalls (failure-mode-like hits,
  by kind or keyword) vs knowledge refs. Next actions are derived from the plan
  graph's ready/first nodes via `next_ready_nodes(graph, set())`.
- `write_rendered`: creates parent dirs and writes UTF-8 text verbatim, so the
  caller controls exact file content.
- Gate visibility helpers treat a non-empty `evidence_contract` as an active
  evidence gate, honouring an explicit `required` flag when present; and read
  `human_review.required` as the human-review gate.
- Determinism: node and edge order follow `graph.nodes`; tests assert identical
  output across two renders for mermaid, markdown, and briefing.
